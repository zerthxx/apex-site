## The one-paragraph mental model

Apex Site is a **Next.js 15/16 App Router monolith** with **Supabase** (Postgres + Auth + Storage) as the entire backend — there is no separate API server. Every page is a React Server Component that queries Supabase directly using the visitor's own cookie-scoped session, so **Postgres Row-Level Security (RLS) policies are the real authorization layer**, not just a backstop. API routes under `src/app/api/` exist for anything that needs a POST/PATCH/DELETE from a client component, or that needs to run with elevated (service-role) privileges — e.g. looking up another user's auth profile, or writing notifications on behalf of someone else. Business data flows in one direction through a pipeline: a public lead (`/api/contact`) becomes a `customers` row, which gets a `quotes` row, which — on acceptance — **auto-creates** a `projects` row, which accumulates `tasks`/`files`/`comments`, which eventually gets an `invoices` row, which gets paid through **Stripe Checkout**, confirmed asynchronously by a **Stripe webhook**.

## Known operational risk (read this before touching the database)

The 9 files under `supabase/migrations/` are **not applied automatically by anything** — no CI step, no Supabase CLI push, no ORM migration runner. Every file says "run this in the Supabase SQL Editor" in its header comment. This project has already been bitten by this twice in this session: quotes were invisible to customers because migration `004` had never been run, and the "Hyväksy" button was broken because migration `009` didn't exist yet. **Whenever you add a new table, column, or RLS policy, you must manually paste the new migration into the Supabase SQL Editor of the production project** — writing the `.sql` file alone does nothing. See [02-database.md](./02-database.md) for the full list of migrations and the drift risks this has already caused.
